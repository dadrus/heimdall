Secret management in heimdall separates the location of secret material from the components that consume it. The former is defined as a secret source and the latter as a secret reference.

== Secret Source Configuration

Secret sources are configured below the `secret_management` section.

[source,yaml]
----
secret_management:
  source_a:
    type: pem
    config:
      path: /etc/heimdall/secrets.pem
  source_b:
    type: inline
    allow_in_rules: true
    config:
      my-api-key: "${API_KEY_FROM_ENV}"
----

Each source has a unique name, a link:{{< relref "#_backends" >}}[backend] `type`, a backend-specific `config`, and optionally an `allow_in_rules` flag (defaulting to `false`) indicating whether this source is allowed to be used from rules.

== Secret References

Required secrets are referenced via `source` and a `selector`

* `source` selects one of the configured secret sources under `secret_management` and is always required.
* `selector` identifies a specific entry within that source. It may be optional for providers exposing exactly one value, such as a PEM certificate bundle.

`selector` is always a provider-local, path-style identifier:

[source,yaml]
----
selector = segment [ "/" segment ]*
segment = string without "/"
----

== Secret Types

heimdall distinguishes between three kinds of secret material.

* **Secret:** A secret represents a single secret value. This can be a string secret, like an API key, a symmetric key, or an asymmetric key. Consumer sides typically define a `secret` property to reference such secret type.

* **Credentials:** Credentials represent structured authentication data, like a username and password, or another structured object. Consumer sides typically define a `credentials` property to reference such secret type. The concrete fields required by such an object depend on the consumer. For example, username/password based consumers expect the selected object to contain a `username` and a `password` field.

* **Certificate Bundles:** A certificate bundle represents a set of certificates used as trust anchors. For example, the JWT authenticator can use a certificate bundle to validate certificates embedded in JWKs. Consumer sides typically define a `trust_anchors` property to reference such secret type.

== Backends

=== PEM Backend

This backend reads PEM encoded files and can expose either asymmetric key secrets or certificate bundles.

PEM files containing private keys are treated as key stores. Certificates in the same file are used to build certificate chains for the matching private keys.

PEM files containing only certificates are treated as certificate bundles and can be used as trust anchors.

To configure this backend as a link:{{< relref "#_secret_source_configuration" >}}[secret source], you have to specify `pem` as type. Configuration is mandatory with the following options:

* *`path`*: _string_ (mandatory)
+
Path to the PEM file.

* *`password`*: _string_ (optional)
+
Password used to decrypt encrypted PKCS#8 private keys.

* *`watch`*: _bool_ (optional)
+
Enables file watching and reloads the provider when the file changes. Optional. Defaults to `false`.
+
NOTE: A PEM file must not change its material kind at runtime. If file watching is enabled, reloads only update the backend if the material kind stays the same. A runtime transition from key material to certificate bundle material, or vice versa, is rejected and the last known good state is kept.

Here an example:

[source,yaml]
----
secret_management:
  my_pem_source:
    type: pem
    config:
      path: /etc/heimdall/secrets.pem
----

For key material, consumers can optionally specify a key id via a selector. If a selector is not specified, the first key entry from the pem file is taken.

When heimdall loads a key store, following algorithm is used to get the key id for the key:

* if the PEM entry with the private key has `X-Key-ID` header specified, this value is used as key id
* Otherwise, if an X.509 certificate is present for the private key, and it has the `Subject Key Identifier` extension set, the hex representation of it is used as key id.
* Otherwise, heimdall calculates the value for the `Subject Key Identifier` according to https://www.ietf.org/rfc/rfc3280.html#section-4.2.1.2[RFC 3280, Section 4.2.1.2] and uses hex representation of it as key id.

Certificate bundles exposed by PEM files do not require a selector.

=== Inline Backend

This backend allows defining secret material directly in the heimdall configuration. It is mainly useful for local development, tests, or deployments where the actual values are injected from environment variables.

To configure this backend as a link:{{< relref "#_secret_source_configuration" >}}[secret source], you have to specify `inline` as type. Configuration is mandatory and contains the values exposed by this backend.

[source,yaml]
----
secret_management:
  my_inline_source:
    type: inline
    config:
      api-key: "${API_KEY_FROM_ENV}"
      redis:
        username: "${REDIS_USERNAME}"
        password: "${REDIS_PASSWORD}"
----

The entries below `config` are exposed by their path-style selector.

For example, the configuration above exposes `api-key` as a string secret and `redis` as a structured credentials object.

NOTE: Avoid putting sensitive values directly into heimdalls configuration file. Prefer using environment variable interpolation as also shown in the example above when using this backend.
